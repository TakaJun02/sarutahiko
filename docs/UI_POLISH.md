# UI 全面ブラッシュアップ仕様（FR-9）

- 版: v0.1（2026-07-13, Fable 起草 — 利用者指示「おしゃれ度 35 点 → 100 点。PC・スマホ両対応」）
- 対象: フロントエンド全画面（ログイン・チャット・サイドバー・ローディング演出の**周辺**）
- 検収方式: **UI 評価専任エージェント**（実装と別）による採点ループ（§5）

## 0. 不可侵の制約（先に読むこと）

1. **LoadingSpinnerV5 の演出ロジック**（morph・Aurora Ring・pending→settled 遷移）は確定仕様
   （docs/UI_LOADING_ANIMATION.md）。色・サイズの微調整は可、動作・タイミングの変更は不可。
2. **ログイン画面の 2 層構造**（ライトエリア＋タイプライター見出し / 黒シート）とタイプライターの
   タイミング（150/75/2000/500ms）は docs/UI_LOGIN.md の確定仕様。構造を壊さず、その内側で磨く。
3. チャットの**中央カラム原則**（AI 回答=中央ドキュメント型 / ユーザー発言=右寄せ吹き出し、
   docs/UI_CHAT.md §1）とセッションサイドバー（§4）の機能要件は維持。
4. SSE スキーマ・store の公開 API・バックエンドは変更しない（フロントのみ）。
5. `prefers-reduced-motion` で装飾モーションを停止（NFR-5）。

## 1. デザインシステム

### 1.1 カラー

- ベースは現行ダーク（#0f1115 系）を継承しつつ、**階調を設計**する:
  `--bg-base`（最奥）/ `--bg-surface`（サイドバー・カード）/ `--bg-raised`（入力欄・メニュー）を
  3 段以上に分離し、白の透過率頼み（white/6, white/8 …）の散発指定を CSS 変数 or Tailwind theme に集約。
- **ブランドグラデーション #FF8A65 → #FFEB3B → #69F0AE** をアプリ全体の「一本の糸」として使う:
  アクセント（アクティブ状態・送信ボタン・カウントダウンチップ・見出しの下線等）に限定し、多用しない。
- ユーザー吹き出しの純白 (#fff) は眩しいので、わずかにトーンを落とすか薄いグラデーションで柔らかく。
- コントラスト: 本文テキストは背景比 4.5:1 以上、補助テキストも 3:1 以上。

### 1.2 タイポグラフィ

- フォントスタック: `"Noto Sans JP", "Hiragino Sans", "Yu Gothic UI", system-ui, sans-serif` を明示
  （外部フォント CDN への依存は追加しない。会場ネットワークが不安定でも崩れないこと）。
- 本文 16px / 行間 1.7〜1.8、見出しは字間 -0.01em 前後。Markdown 描画（見出し・箇条書き・表・コード）
  に prose スタイルを整備し、AI 回答が「読み物」として美しいこと。

### 1.3 形状・立体感・モーション

- 角丸・影・ボーダーのスケールを統一（例: radius 12/16/24px の 3 段。影は「淡い広がり＋1px 内側ボーダー」）。
- マイクロインタラクション: hover/active/focus に 150〜250ms の ease-out トランジション。
  タップ可能要素は押下時にわずかにスケール（0.97 程度）。`focus-visible` リングを全インタラクティブ要素に。
- スクロールバー・テキスト選択色もテーマに合わせる。

## 2. 画面別の磨き込み

### 2.1 ログイン

- タイプライターのカーソルにブランドグラデーション（UI_LOGIN.md §2-1 で許可済み）。
- 入力欄・CTA・エラーパネルの状態デザイン（focus ring、hover、disabled、処理中スピナー）を丁寧に。
- 黒シートに `env(safe-area-inset-bottom)` を確保。ライトエリアの見出しは 390px でも折返しが美しいこと。

### 2.2 チャット本体

- ヘッダー: ガラス質（blur）を維持しつつ、下辺にごく細いブランドグラデーションのヘアラインなど
  「らしさ」を一箇所だけ。
- **空状態（新規会話）をヒーロー化**: アプリアイコン＋挨拶見出し＋
  **カウントダウンチップ**（例: 「🎪 オープンキャンパス2026まであと6日」— 2026-07-19 からクライアント側で算出、
  当日は「本日開催！」、終了後は非表示）＋**おすすめ質問チップ 3〜4 個**
  （タップで入力欄へ。例: 「アクセス方法は？」「模擬講義は何がある？」「学食メニューは？」「無料送迎バスの時刻は？」）。
- 出典: リンクの羅列をやめ、type バッジ（学内ナレッジ / Web）付きの小さなカード or チップに。
- 入力バー: フローティングのピル型を磨く（focus 時に淡い brand-mint グロー、textarea 自動リサイズ、
  送信ボタンは有効時のみ精彩）。`safe-area-inset-bottom`・`100dvh` ベースでモバイルの被りを解消。

### 2.3 サイドバー（FR-7 実装済みのものを磨く）

- 項目の hover/active をピル型ハイライトで。アクティブスレッドはブランドアクセントの细いバー等で明示。
- モバイルドロワーはオーバーレイのフェード＋パネルのスライドを 200ms 前後で滑らかに。
- 最下部のユーザーカード（アバター円＝イニシャル、名前、ログアウト）を整える。

### 2.4 レスポンシブ規律

- ブレークポイント: モバイル（<768px）/ タブレット（768〜1023px）/ PC（≥1024px）で全画面を確認。
- タップターゲット 44×44px 以上。モバイルでの文字サイズ・余白の詰まりすぎに注意。
- 390×844（iPhone 系）・1440×900 を一次基準とする。

## 3. 実装ノート

- 変更は frontend/ のみ。Tailwind theme（tailwind.config.js）と style.css にトークンを集約し、
  コンポーネント内のマジックナンバー指定を削減する。
- 依存パッケージの追加は原則なし（アイコンは inline SVG で。アニメーションライブラリ禁止）。
- `npm test` と `npm run build` が通ること。

## 5. 評価ループ（検収方式）

1. 実装エージェントが本仕様で改修 → `AGENT_MODE=mock` のバックエンド＋Vite dev でアプリを起動。
2. **評価エージェント（実装と別人格）**が Playwright で
   390×844 / 768×1024 / 1440×900 の 3 幅 × 主要状態（ログイン・空チャット・会話中・ストリーミング中・
   サイドバー/ドロワー開）を実際に操作・撮影し、§4 ルーブリックで採点。減点は「観点・画面・具体箇所・
   修正提案」の形式で列挙する。
3. **90 点以上で合格**。未満なら指摘を実装エージェントに差し戻して再実装（最大 3 ラウンド）。
4. 最終ラウンドの採点表とスクリーンショット所見を PR 説明（または報告）に残す。
