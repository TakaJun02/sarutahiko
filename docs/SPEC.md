## 1. 目的

秋田県立大学 本荘キャンパスのオープンキャンパス 2026 来場者（主に高校生・保護者）が、
学部・学科、施設、アクセス、当日のイベント等について自然言語で質問すると、
学内ナレッジ（RAG）と Web 検索を組み合わせて正確に回答する AI エージェントを提供する。

## 2. 想定ユーザーと利用シーン

- **主**: オープンキャンパス来場者（高校生・保護者）。スマートフォンからの利用が中心（参考動画も 738×1454 の縦画面）。
- AI の名称（2026-07-14 追加・利用者指示、FR-13）: **APU-Navi（エーピーユー・ナビ）**。
  UI の表示名およびシステムプロンプトのペルソナ名として用いる（表記は「APU-Navi」で固定）。
- 回答トーン（2026-07-13 改訂・利用者指示）: **オープンキャンパスの明るい学生ガイド**のペルソナ。
  来場者がわくわくするような元気で親しみやすい口調（「〜ですよ！」「ぜひ体験してみてください！」等）。
  高校生にも分かる語彙。既定言語は日本語。
  - 絵文字は 1 回答 0〜3 個まで（見出し・箇条書きの構造を壊さない範囲）。
  - 盛り上げるのは言い回しのみ。事実（数字・固有名詞・日程）は誇張・改変しない（FR-1 の正確性ルールは全て維持）。
- 誤情報が進路選択に影響しうるため、**根拠（出典）を持たない断定をしない**こと。

## 3. 機能要件

### FR-1 チャット応答（Agentic RAG）
- ユーザーの質問に対し、エージェントが以下を自律的に判断して回答する:
  1. 質問の解釈・必要な情報の特定
  2. 学内ナレッジ（ベクトル検索）からの取得
  3. ナレッジで不足する場合の Web Search（例: 最新のイベント日程）
  4. 取得情報の十分性評価（不足なら再検索）
  5. 回答生成（出典付き）
- ナレッジにも Web にも根拠がない場合は「分からない」と答え、問い合わせ先（大学公式サイト等）を案内する。
- **丸投げ回答の禁止（2026-07-11 追加・利用者指示）**: 「公式サイトや当日の紹介でご確認ください」の類いを
  回答の主内容にしてはならない。調べれば答えられる内容はハーネス側で RAG・Web Search を粘り強く回して
  具体的に回答する（詳細: `docs/AGENT_HARNESS.md`）。丸投げが許されるのは検索を尽くしても根拠ゼロの場合のみで、
  その場合も何を調べたかを述べる。

### FR-2 推論ステップの実況通知
- 推論の各ステップで「今何をしているか」を短い日本語テキストでフロントに随時配信する
  （例: 「質問を分析しています…」「学内ナレッジを検索しています…」「Web で最新情報を確認しています…」「回答をまとめています…」）。
- 配信は SSE で行う。イベントスキーマは `docs/ARCHITECTURE.md` に定義。
- フロントはこのテキストをローディング演出の横に表示する（`docs/UI_LOADING_ANIMATION.md`）。

### FR-3 回答のストリーミング表示
- 回答本文はトークンストリーミングでフロントに配信し、逐次描画する。
  （参考実装の「単語ごと 40ms 表示」は日本語と相性が悪いため、実ストリーミングに置き換える。）
- 回答は Markdown としてレンダリングする。

### FR-4 会話の継続
- スレッド（会話履歴）を保持し、文脈を踏まえた応答を行う。
- エージェントのコンテキストに含める履歴は**直近 4 ターン（= 8 メッセージ）**
  （2026-07-14 改訂・利用者指示、FR-18-5。従来は実質 2 ターン）。

### FR-5 ローディング演出
- Ver1.0: guidanceLLM2 の演出を完全再現（詳細仕様: `docs/UI_LOADING_ANIMATION.md`）。
- Ver5.0: Ver1.0 を土台に大幅アップデート（方針は同ドキュメント。詳細仕様は Fable が別途起草）。

### FR-6 ログイン（2026-07-11 追加、2026-07-13 改訂）
- パスワードなしのニックネーム登録制ログイン（詳細仕様: `docs/UI_LOGIN.md`）。
- **属性（高校生/保護者/その他）は 2026-07-13 に廃止（利用者指示）**。登録はニックネームのみ。
  API の `role` フィールド・`users.role` カラム・ロール別トーン指示も削除する（既存 DB はマイグレーションで rebuild）。
- ログイン画面は ChatGPT / Gemini アプリを参考にしたスタイリッシュな UI。
- セッショントークンで API を保護し、会話スレッドをユーザーに紐づける。

### FR-7 会話履歴の永続化とセッションサイドバー（2026-07-13 追加）
- スレッドは SQLite に永続化済み（FR-4）。これに加え、**画面リロード後も会話が復元**されること。
- ChatGPT/Gemini アプリ同様、**左サイドバー**に自分のスレッド一覧（updated_at 降順）を表示し、
  任意のスレッドへいつでも切り替えられる。
- 「新しい会話」の開始・スレッド名の変更・スレッドの削除ができる。
- URL を `/chat/:threadId?` とし、リロードで同じスレッドが開く（新規会話は最初の回答完了時に URL を置換）。
- PC（≥1024px）は常設サイドバー、モバイルはハンバーガー → ドロワー表示。

### FR-8 日時・イベント認識メッセージ（2026-07-13 追加）
- バックエンドが **JST の現在日時**から「時間コンテキスト」を**決定論的に**生成し、生成プロンプトへ注入する
  （LLM に日付・残り時間の計算をさせない）。
- オープンキャンパス2026 = **2026-07-19（日）9:30〜15:00・本荘キャンパス**
  （出典: knowledge/event-open-campus-2026-systems.md、確度 high）。
- 開催前は「オープンキャンパスまであと N 日！」、当日は「今は〇〇が開催されています」
  「あと M 分後に〇〇が始まります」、終了後はお礼、を回答へ自然に織り込めること。
- 織り込みは**任意**であり、毎回答に必ず含める必要はない（時間の話題が自然に絡むときのみ）。
  含める場合は**回答の末尾**に添えること。**回答の冒頭には置かない**（2026-07-13 利用者指示）。
- 当日のイベント時刻は検証済みスケジュール `backend/app/data/oc2026_schedule.json` のみを根拠とする。
  スケジュールにないイベントの時刻を推測して述べてはならない。

### FR-10 出典の折りたたみ表示（2026-07-13 追加）
- 回答メッセージ下部の「出典」セクション（`frontend/src/views/ChatView.vue` の `message.sources` 表示部）を**折りたたみ可能**にする。
- **デフォルトは折りたたんだ状態**。ユーザーが見たいときだけ展開する。
- 折りたたみ時はヘッダ行のみ表示し、「出典」ラベル＋**件数**（例: `出典 3件`）＋開閉を示すシェブロンアイコンを置く。
- 要件:
  - ヘッダ行はボタン（`<button type="button">`）とし、クリック/タップで開閉をトグル。タップ領域は既存 UI と同じく **min-height 44px** を確保（FR-9 の検収基準を踏襲）。
  - `aria-expanded` を付与し、シェブロンは開閉状態に応じて回転（`prefers-reduced-motion` 時はアニメーションなしで即時切替 — NFR-5）。
  - 開閉状態はメッセージごとに独立。ストリーミング完了で sources が到着した時点でも**閉じたまま**表示する。
  - 既存の出典チップ（knowledge/web バッジ・外部リンク挙動）は展開時にそのまま表示。デザインのトーン（色・角丸・トランジション）は既存 UI に合わせる。
  - 出典が 0 件のときはセクション自体を表示しない（現行どおり）。

### FR-11 経路案内（2026-07-13 追加・利用者指示、2026-07-14 改訂）
- 「〇〇はどこ」「〇〇への行き方」「どの部屋」等の**場所・経路質問**に、棟・ゾーン・階・部屋番号まで
  具体的なテキストで回答できること（詳細仕様: `docs/ROUTE_GUIDANCE.md`）。
- **学内の地点間経路案内が主要件**（2026-07-14 利用者明確化。例:「食堂から D404 までの経路」）。
  公式図面から Fable が裁定した経路グラフ（連絡通路の階・棟のつながり）を根拠に、
  経由地の列挙＋階の指定で案内する（ROUTE_GUIDANCE.md §7）。
- 実現はナレッジ駆動（キャンパスマップ・OC2026 会場インデックス・徒歩経路の各ナレッジ整備）＋
  analyze/generate プロンプトの微修正のみ。エージェントのグラフ構造は変更しない。
- 根拠にない道順（曲がる方向・徒歩分数等）を創作しない。場所の特定まで案内した上で、
  細部は棟内案内表示・当日スタッフへ誘導してよい（FR-1 の丸投げ禁止には当たらない）。
- 地図 UI・ナビゲーション画面は引き続きスコープ外（§5。あくまでチャットのテキスト回答）。

## 4. 非機能要件

- **NFR-1**: LLM 推論はローカル GPU（RTX 3090 Ti 24GB）上の vLLM のみを使用。外部 LLM API に依存しない。
- **NFR-2**: 最初のステータス通知は送信後 1 秒以内に返す（体感応答性）。
- **NFR-3**: スマートフォン縦画面を第一とするレスポンシブ UI。
- **NFR-4**: docker compose で全コンポーネントを起動できること。
- **NFR-5**: `prefers-reduced-motion` を尊重する（Ver5.0 演出で必須）。

## 5. スコープ外（現時点）

- パスワード等の資格情報管理・本人確認（FR-6 の軽量ログインは行うが、それ以上の認証強度は求めない）
- 個人情報の収集（ニックネームのみ。本名・連絡先は扱わない。属性も 2026-07-13 に廃止）
- 多言語対応（構造上は拡張可能にしておくが、初期リリースは日本語のみ）
- 地図ナビゲーション機能（guidanceLLM2 の GUIDANCE MAP 相当は対象外。あくまでチャット。テキストでの経路案内は FR-11 で対応）
