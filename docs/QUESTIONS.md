## Q-011: 生成プロンプト予算超過時に残す最低根拠量
- 日付: 2026-07-12
- 起票: Codex
- 状況: `docs/AGENT_HARNESS.md` §5 ルール6は「会話履歴を常時保持し、根拠コンテキストを縮小して再組み立て（コンテキスト予算 = プロンプト予算 − system − 履歴 − user骨格）」と定めているが、履歴と user 骨格だけで予算を消費する場合に最低何件・何文字の根拠を必ず残すかは未定義。暫定実装では、上記式で算出した根拠コンテキスト予算を厳守し、0 以下なら根拠は空にしても会話履歴を保持する。根拠側の詰め順は §5 の優先順（grepヒット → Web → ベクトルのみ）に従い、入る範囲で本文を切り詰める。
- 裁定 (Fable): **最低根拠量を保証する**（2026-07-12）。理由: 現行既定値（窓 2816・回答 640・マージン 192 → プロンプト予算 1984 推定トークン）では、履歴最悪ケース（4件×500字 ≈ 推定1500トークン）＋ system＋user 骨格で予算を超え、根拠 0 で generate する事態が現実に起こり得る。調査した根拠を捨てて回答するのは本末転倒のため、以下とする:
  1. generate 時の根拠コンテキストに最低 `MIN_GENERATION_CONTEXT_TOKENS = 384`（推定トークン）を確保する。
  2. 残余コンテキスト予算が 384 を下回る場合、履歴の**件数は維持**したまま各メッセージの文字上限を段階的に縮める（500 → 250 → 120 字。下限 120 字）。履歴メッセージの削除・件数削減は不可（ルール6の常時保持と両立させる）。
  3. 下限 120 字まで縮めても 384 に届かない場合は、その時点の残余予算で妥協してよい（根拠 0 も許容）。
  4. evaluate モードのプロンプト構築は対象外（現状維持）。強制コンテキスト予算パス（実トークン再構築・リトライ）にも 1〜3 を適用する。
- 裁定補足 (Fable, 2026-07-12 検収時): 強制コンテキスト予算パスでは、**実測由来の予算を単純に 384 まで押し上げてはならない**（押し上げると再構築後のプロンプトが縮まらず、実コンテキスト超過から回復できない）。強制パスでの 384 確保は「履歴の文字上限短縮（500→250→120）で捻出した推定トークン分を強制予算に加算する」方法のみ許可する。つまり最終コンテキスト予算 = min(その段階の残余推定予算, 強制予算 + 履歴短縮で浮いた推定トークン)。384 に届かなければ届く段階まで短縮し、下限 120 字でも届かなければその値で妥協する。
