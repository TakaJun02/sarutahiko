## Q-002: `/api/chat` のリクエスト本文フィールド名が未定義
- 日付: 2026-07-11
- 起票: Codex
- 状況: `docs/ARCHITECTURE.md` §3 は「質問と `thread_id` を受ける」としているが、質問本文の JSON フィールド名が明記されていない。暫定的にフロントは `message` を送信し、バックエンドは互換性のため `message` と `question` の両方を受け付ける実装にした。
- 裁定 (Fable): **正式フィールド名は `message` とする**（ARCHITECTURE.md に明記済み）。`question` の互換受け付けは不要なので Phase 2 で削除すること（2026-07-11）。

## Q-003: `done.sources` の「実際に使った出典」の判定方法
- 日付: 2026-07-11
- 起票: Codex
- 状況: Phase 2+3 タスクでは `done` イベントに「sources actually used」を載せる指定があるが、ストリーミング生成後に LLM が本文中で実際に参照した根拠を機械的に確定する仕様は未定義。暫定的に、回答生成プロンプトへ投入したナレッジチャンクおよび Web 検索結果を「使用出典」とみなし、knowledge は `title` + 最初の `source_urls[0]`、web は `title` + `url` を重複排除して返す実装にした。
- 裁定 (Fable): **暫定判断を承認**（プロンプト投入コンテキスト＝使用出典とみなす方式が実用上妥当）。本文中の実引用判定は過剰工学なので導入しない。ただし evaluate ステップで「不十分」と判定され generate に使われなかったチャンクは含めないこと（2026-07-11）。

## Q-004: 検索クエリ加工と関連度閾値の統合チューニング（Fable 起票・裁定）
- 日付: 2026-07-11
- 起票: Fable（統合テストでの実測に基づく）
- 状況: 検索クエリに「秋田県立大学 本荘キャンパス」を付加すると、全文書がキャンパス関連のため無関係な質問（天気・学長等）でもスコアが約 0.6 まで底上げされ、evaluate の不十分判定（閾値 0.2）が機能しなかった。図書館の質問で図書館文書がトップ3に入らない retrieval 劣化も確認。
- 裁定 (Fable): 付加を廃止し素の質問文で検索、`min_relevance_score` を 0.45 に変更（実測: 関連質問トップ ≥0.58、無関係 ≤0.39）。`backend/app/agent/graph.py` に直接反映済み。Web 検索クエリの付加は維持（2026-07-11）。
