## 2. ファイル形式

`knowledge/` 配下に 1 トピック 1 Markdown ファイル。frontmatter 必須:

```markdown
---
id: faculty-systems-science          # kebab-case 一意 ID
category: faculty | facility | access | admission | campus-life | event | career | area | lab
title: システム科学技術学部の概要
source_urls:
  - https://www.akita-pu.ac.jp/...
retrieved_at: 2026-07-11             # Codex が取得した日付
confidence: high | medium | low      # 情報源の確からしさ
---

本文（日本語）
```

### 本文の書き方（v0.2 で全面改訂・厳守）

1. **1 ページ = 1 ファイルの全文書き起こし**。対象ページに書かれている情報を、
   **端から端まで 1 つも漏らさず**日本語で書き起こす。**要約・省略・「詳細は〜参照」への置き換えは禁止**。
2. 表はMarkdown表として、箇条書きは箇条書きとして、数値・日付・固有名詞・注記・例外事項はすべてそのまま転記する。
3. 文字数の上限は設けない（チャンク分割はインジェスト側が見出し単位で行うため、
   **論理的な見出し（##）でセクションを区切る**こと）。
4. 非常に長いページは `<id>-part2` のように分割してよい（frontmatter は同じ category、title に (2/2) 等を付す）。
5. ナビゲーション・フッター等の定型文は除外してよいが、本文に関わる情報は一切除外しない。

### 2.1 検索語彙の自己完結性（v0.5 追加・厳守）

検索（ベクトル・字句とも）はファイル単体ではなく**セクション（`##` 見出し）単位のチャンク**で行われる。
「ページを見れば分かる」文脈はチャンクに残らないため、次を必須とする:

1. **本文冒頭に導入文を必ず書く**: そのページが「何の・誰の・何についての」情報かを、
   **対象の正式名称・通称・別名を併記**した日本語 1〜2 文で書く。
   例（lab-cps-members.md）:
   「秋田県立大学 経営システム工学科 サイバーフィジカルシステム研究室（CPS研、山口研究室）の
   メンバー（教員・大学院生・学部生・修了生）の一覧である。」
2. **見出しは日本語を主とする**: 原文が英語見出し（Members, Access 等）でも、
   日本語を主・英語を括弧書きにする（例: `## 学生メンバー（Members）`）。
   来場者の質問語彙（メンバー・学生・アクセス・行き方 等）が見出しか導入文に現れること。
3. **title にも正式名称を含める**: `CPS研究室 Members` のような略称+英語のみは不可。
   例: `サイバーフィジカルシステム研究室（CPS研） メンバー一覧`。
4. **表だけのセクションを作らない**: 表の直前に「以下は〜の一覧」の 1 文を置く
   （チャンク分割後もそのチャンク単体で意味が取れるように）。
   **前置文は検索語彙を運ぶこと（v0.5.1 精緻化・2026-07-12 検収で追加）**:
   「以下は『修士2年』に関する一覧である」のような見出しの反復は不可。
   対象の正式名称と、想定質問の語彙（学生メンバー・氏名・研究テーマ 等）を含めて書く。
   例: 「以下はサイバーフィジカルシステム研究室 修士課程 修士2年の学生メンバーの氏名と研究テーマの一覧である。」
   （実測: 形式的な前置文では『学生メンバー』質問で修士の氏名表チャンクがベクトル・grep 両方から漏れた）
   また、見出しだけのセクション（例: `## 修士課程` 直下に本文なしで `###` が続く）には
   1 文の説明を補い、空セクションのチャンク化を避ける。
5. 想定質問の中核語（出展・展示・模擬講義・メンバー・学費・日程・アクセス 等のうち
   該当するもの）が本文に自然に含まれるよう書く（§1.2 の出展ガイドルールの一般化）。

**既存ファイルの総点検（v0.5・Codex 実施）**: 全ナレッジを上記 1〜4 で点検・改修する。
最低限、実測で問題が確定している `lab-cps-members.md` と、英語見出しの `lab-cps-*` 各ファイルは必須。
改修後は再インジェスト（ハーネス v5 §V5-1 の再作成と同時でよい）。

## 3. 収集・記述ルール（Codex 厳守）

1. **一次情報源を優先**: 大学公式サイト > 自治体・公的機関 > その他。まとめサイト・受験ブログは原則使わない。
2. **出典 URL を必ず記録**（`source_urls`）。URL のない情報は書かない。
3. **数値・固有名詞は出典の記載どおり**に書く。推測で補完しない。曖昧なら `confidence: low` を付けて本文にもその旨を書く。
4. 年度依存の情報（入試日程・オープンキャンパス日程等）は**年度を本文に明記**する。
   - **オープンキャンパス情報は特に厳格に**（v0.3 追記・利用者指示）: 本システムは「オープンキャンパス **2026**」に特化している。過年度（2025 以前）のオープンキャンパス情報は原則採録しない。参考として採録する場合は、**title・本文の双方に開催年度を明記**し、2026 年度の情報と混同されない書き方にする（「今年」「例年どおり」等の年度が曖昧になる表現は禁止）。
5. 古い情報しか見つからない場合は、取得できた年度を明記した上で採録し、`confidence: medium` 以下とする。
6. 収集のたびに `knowledge/SOURCES.md` に「カテゴリ / ファイル / 主要 URL / 取得日」を追記する（検収用台帳）。

## 4. インジェスト（Phase 3〜4 で実装）

- `knowledge/ingest/` のスクリプトが Markdown を読み、frontmatter をメタデータとして
  埋め込みモデル（v0.5 以降: `Qwen/Qwen3-Embedding-8B`・リモート vLLM。ARCHITECTURE.md §5.2）で
  埋め込み、Qdrant に投入する。埋め込み対象は `title + heading + 本文`（ハーネス v5 §V5-1）。
- チャンク方針: 見出し単位で分割、最大 ~500 トークン、オーバーラップ 50。
- `category` と `confidence` は Qdrant のペイロードに載せ、検索フィルタと回答時の出典表示に使う。
