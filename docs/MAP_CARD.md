## 1. 目的とスコープ

FR-11（経路案内）のテキスト回答に、キャンパス模式図の**マップカード**を添えて視覚化する。
あわせて、出発地が特定できない経路質問に対しエージェントが**マップタップカード**
（現在地の申告 UI）を提示できるようにする。

- **対象**:
  - (a) 経路質問（FR-11 (c)）への**経路カード**（mode: `route`）: 経由棟のハイライト＋階バッジ＋ステップリスト
  - (b) 場所質問（FR-11 (b)）への**場所カード**(mode: `place`): 目的地の棟ハイライト＋階バッジ
  - (c) 出発地不明の経路質問への**マップタップカード**（mode: `ask_origin`）: タップで現在地を申告
- **対象外**:
  - GPS・BLE 等による現在地の自動取得（今回のリソースでは非現実的。現在地は申告制で解決する）
  - 地図ナビゲーション画面（SPEC §5 のスコープ外は維持。本カードは**チャット回答に添える静的な模式図**であり、
    ナビ画面・追従表示・拡大縮小操作は持たない。SPEC §5 に例外裁定を追記済み）
  - QR チェックポイント（`?loc=` パラメータによる現在地プリセット）は将来拡張とし、本 FR に含めない。
    ただし §5 の設計はこれを妨げない形にする。

## 2. 方式裁定（Fable 決定）

### 2-1. マップタップは「エージェント起点・ターン終端 elicitation」方式

利用者確認済み（2026-07-17）: マップタップの要求はユーザー入力ターンではなく、
**エージェントが必要と判断したタイミングで発火**する。実現方式は次のとおり裁定する。

- エージェントは analyze の結果「経路質問だが出発地が特定できない」場合、検索・生成へ進まず
  **そのターンを短く終える**: 定型の短文（§6-3）を token として流し、`map`
  イベント（mode: `ask_origin`）を送って `done` する。
- ユーザーがカードのノードをタップすると、フロントが**通常の `POST /api/chat`** で
  元の質問文＋`origin_node` を送信し、新しいターンとして経路回答が返る
  （v0.3 改訂・§11-2。「現在地は〇〇です。＋元質問」への合成は**バックエンド内部**で行い、
  UI 上は現在地チップとして表示する — ユーザーから見えるのは「聞かれたから答えた」の 1 操作のみ）。
- **実行中の LangGraph run を中断・保留して再開する方式（mid-run interrupt）は採用しない**。理由:
  1. 来場者はタップせず立ち去り得る。run と SSE 接続を無期限に保持するとリークし、
     本番 Nginx の `proxy_read_timeout 300s` でどのみち切断される。
  2. 検収済みハーネス v5 に checkpoint/resume 機構はなく、追加は回帰リスクが大きい。
  3. ターン終端方式でも体験は同一（エージェントが尋ね、マップが現れ、タップで会話が続く）。
     合成メッセージは既存の FR-11 出発地抽出（「食堂から D404」）のコードパスをそのまま通るため、
     **リクエスト body・DB スキーマの変更が不要**になる。

### 2-2. 階層の表現は「平面パス＋階バッジ＋ステップリスト」

利用者指摘（学内は階層がある）への裁定（2026-07-17 承認済み）:

- 平面 SVG は**水平経路（棟のつながり）だけ**を描く。
- 階の情報は (1) エッジ・ノード上の**階バッジ**（例: 連絡通路上に「4F」、目的地に「D414・4階」、
  入口に「2階が地上」）、(2) カード下部の**ステップリスト**（垂直移動は必ずここに書く）で表現する。
- マップタップ（現在地申告）は**棟単位**とし、階の選択は求めない（棟内の垂直移動は経路の形を変えないため。
  9 ノードのシンプルさを優先する）。

### 2-3. 経路はバックエンドの決定的計算。LLM を関与させない

FR-11 §2 の原則（12B に空間推論をさせない）を踏襲する。カードの経路（経由ノード・使用エッジ・ステップ文）は
`docs/ROUTE_GUIDANCE.md` §7.2 の確定経路グラフから**コードで決定的に計算**し、LLM の生成文とは独立に作る。
テキスト回答は従来どおり RAG（route-graph ナレッジ）由来であり、両者は同一の §7.2 を正とするため矛盾しない。

- `knowledge/event-oc2026-location-index.md` の表から、**棟（ゾーン）が埋まっている行のみ**を

- 施設名の別名表（食堂/学食/カフェテリア → `cafeteria`、総合受付/受付 → `k`、

## 4. 経路計算とステップ生成（backend・決定的）

### 4-1. 経路探索

- §3-2 の重み付き無向グラフ上の Dijkstra。出発ノード・目的ノードを受け取り、
  ノード列とエッジ列を返す。
- **並行エッジ（g1–d の E1/E2、g2–d の E3/E4）の選択**: 目的地の階が判明している場合は
  連絡通路 `floor` が目的階に一致するものを優先（例: D414=4階 → E1）。不明な場合は
  現在の入館階（= groundFloor）に近いものを優先（→ E2/E4）。
- 主経路の整合: FR-11 検収時の裁定（K→学部棟Ⅱ系は E6 前面通路の直行が主経路・連絡通路 2 本の
  屋内迂回は代替）を重みで再現できていることを検収で確認する（§9-6）。

### 4-2. ステップリスト生成（垂直移動はここが担当）

ノード列＋エッジ列から次の規則で日本語ステップ文を機械生成する（LLM 不使用・テンプレート）:

1. 出発: 「◯◯を出る」（出発ノードが cafeteria で階別名（図書館=3階）の場合はその階から）
2. 移動エッジごと: 種別が徒歩/通路なら「◯◯方面へ（徒歩 約N分）」（`minutes` がある場合のみ分数付き）
3. 入館: 対象ノードの `groundFloor != 1` なら「◯◯は {gf}階が地上・{gf}階から入る」
4. 連絡通路エッジの手前: 現在階 ≠ `floor` なら「エレベーターまたは階段で {floor}階へ」を挿入し、
   「{floor}階の連絡通路で ◯◯棟へ」
5. 到着: 部屋の階が判明していて**現在階と異なる**なら「エレベーターまたは階段で {階}階へ → {部屋番号}」、
   判明していて現在階と同じなら「{部屋番号}（{階}階）へ」（冗長な垂直移動ステップを出さない。v0.2 精緻化）、
   不明なら「{部屋番号} の階は館内表示・当日スタッフでご確認ください」
6. **曲がる方向（右折・左折）は生成しない**（FR-11 の反捏造ルール）

## 5. SSE スキーマ拡張（`event: map`）

既存イベント（status/token/done/error）は**不変**。`POST /api/chat` のリクエスト body には
v0.3（FR-27）で任意フィールド `origin_node` を追加する（§11-2。省略時は従来と完全同一挙動）。
新イベント `map` を追加する（ARCHITECTURE.md §3 に反映済み）:

```
event: map
data: {
  "mode": "route" | "place" | "ask_origin",
  "origin":      {"node": "cafeteria", "label": "カフェテリア（食堂）"} | null,
  "destination": {"node": "d", "label": "大学院棟", "room": "D414" | null, "floor": 4 | null} | null,
  "path":  {"nodes": ["cafeteria","g1","d"], "edges": ["E6a","E1"]},   // mode=route のみ
  "steps": ["カフェテリアを出る", "学部棟Ⅰへ", "..."],                  // mode=route のみ
  "prompt": "いまいる場所をマップでタップしてください",                  // mode=ask_origin のみ
  "question": "<元の質問文（タップ時の再送用）>"                          // mode=ask_origin のみ
}
```

- 送信タイミング: いずれの mode も **token 送信完了後・`done` の直前に 1 回**。
- 送らない場合（経路グラフ外の目的地・通常質問）は従来と完全に同一のストリームとなる（後方互換）。
- 将来拡張（QR `?loc=`）はこのスキーマに影響しない（現在地の受け渡しは §7-3 の合成メッセージ方式のため）。

## 6. エージェント変更（graph.py・追加のみ）

グラフ構造（ノード・エッジ・ルーティング）は原則不変。追加点は 3 つ。

### 6-1. analyze 出力の拡張

analyze の JSON 出力に `route` フィールドを追加:

```
"route": {"type": "route" | "place" | null,
          "origin": "<ユーザー表現のまま>" | null,
          "destination": "<ユーザー表現のまま>" | null}
```

- `type=route`: 地点間の行き方を聞いている（「〜から〜へ」「〜への行き方」）。
  `origin` は**質問文または直近 4 ターン履歴から**分かる場合のみ埋める（推測しない）。
- `type=place`: 場所を聞いている（「〜はどこ」）。
- システムプロンプトへの追記は FR-11 §4.1 の様式に合わせ、AGENT_HARNESS.md に §V5 追記する。
  **「追記」の意味（v0.2 明確化）**: 検収済みの既存プロンプト文言（役割宣言・keywords 指示・例・出力形式）は
  **一字も変更せず**、route 抽出の指示文と JSON 形式の `route` フィールドのみを末尾に加える。
  既存文言の圧縮・言い換えは検索品質の無言リグレッションになるため禁止。

### 6-2. 解決（resolver・コード）

`origin` / `destination` の文字列を §3-1 別名表と §3-3 `locations.json` で
ノード（＋部屋・階）に解決する。解決は NFKC + casefold 正規化の辞書引きのみ（LLM 不使用）。

### 6-3. 分岐（ターン終端 elicitation とカード付与）

- **`type=route` かつ destination 解決済み かつ origin 未解決** →
  検索・生成をスキップし、`status`（step: `generate`、text: 「現在地を確認しています…」）→
  定型短文を token で送出（例: 「いまいる場所をマップでタップして教えてください！
  そこからの行き方をご案内します🗺️」）→ `map`（mode: `ask_origin`、`question`=元質問）→ `done`。
  このターンも通常どおりスレッドに保存する。
- **`type=route` かつ 両端解決済み** → 従来フローで回答生成し、`done` 直前に
  `map`（mode: `route`、§4 の計算結果）を付与。
- **`type=place` かつ destination 解決済み** → 従来フロー＋ `map`（mode: `place`）。
- **上記以外（解決不能・type=null）** → 従来フローのまま。カードなし。
  誤爆（経路質問でないのに ask_origin）を避けるため、**destination が解決できない限り
  ask_origin は絶対に出さない**。
- 学外アクセス質問（FR-11 (a)。駅・空港・車）は destination がノード解決されないため
  自動的に対象外となる（従来どおりテキストのみ）。

## 7. フロントエンド（MapCard コンポーネント）

### 7-1. 表示（mode: route / place）

- アシスタントメッセージの本文下・出典（FR-10）より上にカードとして描画。
- FR-25 のなめらか文字送りと干渉しないよう、**カードの表示は文字送り完了後**
  （sources と同じタイミング系。reduced-motion 時は即時）。
- 経路ハイライトは path のエッジ・ノードを強調。連絡通路エッジには `floor` バッジ（例: 「4F 連絡通路」）、
  `minutes` があるエッジには「約10分」バッジ、目的地ノードには部屋・階バッジ（例: 「D414・4階」）、
  `groundFloor != 1` の経由棟には「2階が地上」注記。
- カード下部に `steps` を番号付きリストで表示。折りたたみ可（初期状態は展開。FR-10 の開閉様式を踏襲）。
- 履歴リロード時も再現できるよう、`map` payload はメッセージのメタとして SQLite に保存し、
  `GET /api/threads/{id}` のメッセージに含めて返す（フロントは受信時と同じ描画をする）。

### 7-2. マップタップ（mode: ask_origin）

- 9 ノードすべてをタップ可能として描画（タップ領域は **min 44px**。FR-9 基準）。
- アクセシビリティ: SVG タップと等価な**ノード名ボタンのチップ列**をカード下部に併記する
  （スクリーンリーダー・キーボード操作対応。`aria-label` 必須）。
- カードは**最新のもののみ活性**。タップ後、または利用者が別のメッセージを送った後は
  非活性（淡色化・タップ無効）にする。履歴リロード後は常に非活性。

### 7-3. タップ時の送信（v0.3 で改訂 → §11-2 が正）

- v0.1 の「合成メッセージをユーザーバブルにそのまま表示する」方式は FR-27 で廃止。
  タップは `origin_node` フィールド付きの通常送信となり、UI には**現在地チップ**が残る（§11-2）。
- 「合成文が会話ログ（DB content）に残り、4 ターン履歴を通じて以後のターンでも出発地として
  自然に参照される。サーバー側に現在地状態を持たないため鮮度切れが構造的に起きない」という
  v0.1 の設計効果は、合成をバックエンド内部処理に移しても**そのまま維持される**。

### 7-4. 出発地の前提明示

origin が履歴由来（今回の質問文に明記されていない）の場合、回答文頭で前提を明示する
（例: 「カフェテリアからの経路をご案内します！」）。generate プロンプトのルール 8 に 1 文追記:

> 「経路の出発地が直前の会話から引き継いだものである場合、回答の冒頭でその出発地を明示してください。」

## 8. 非機能・制約

- カードはスマホ縦画面（NFR-3）で横スクロールなしに収まること（SVG は viewBox でスケール）。
- `prefers-reduced-motion` 時はハイライトのアニメーションを行わない（NFR-5）。
- ask_origin ターンは LLM 生成を伴わないため、送信から `done` まで 2 秒以内を目安とする。
- 既存検収済み挙動の回帰禁止: `map` イベントを一切送らないケース（通常質問・学外アクセス）で
  ストリーム内容がバイト単位で従来と同一であること（テストで担保）。

## 11. FR-27 改訂仕様（2026-07-17 利用者指示・v0.3。§2-1/§7-2/§7-3 より本節が優先）

### 11-1. ask_origin 受付中の composer ロック

- ask_origin カードが**活性**（`mapInteractive === true`）である間、composer を送信不能にする:
  textarea・送信ボタン・例文チップ・Enter 送信のすべてを無効化（`isSending` 中と同等の扱い）。
  placeholder は「マップから現在地を選んでください」に切り替える。
- ロックの解除条件は次の 3 つのみ:
  1. ノードのタップ（→ §11-2 の送信が始まり、以後は通常の `isSending` 制御へ）
  2. カードの**キャンセル**（下記）
  3. スレッド切替・リロード（履歴復元でカードは常に非活性 = ロックなし。§7-2 の既定どおり）
- **キャンセル余地（Fable 裁定）**: カード下部に控えめなテキストボタン
  「現在地を選ばずに続ける」を置く。タップでカードを非活性化しロックを解除する（送信はしない）。
  来場者が経路を諦めて別の質問に移りたいときの脱出口であり、
  「受付中はフォームに戻れない」という本項の原則は保ったまま**受付そのものを終了**させる操作である。
- タップ送信が**失敗**した場合（ネットワークエラー等）: エラーメッセージは通常どおり表示しつつ、
  カードを再活性化してロックを維持する（再タップで再試行できる。retry 経路でも `origin_node` を保持）。

### 11-2. タップ＝フロー内の現在地申告（`origin_node` フィールド方式）

「タップしたら新しい自然言語クエリが投稿された」ように**見せない**。位置付けは
「エージェントが処理の一連の流れの中で必要だから聞き、ユーザーが答えた」である。

- **リクエスト**: `POST /api/chat` body に任意フィールドを追加 —
  `{"message": "<ask_origin カード payload の元質問>", "thread_id": "...", "origin_node": "<ノード id>"}`。
  `origin_node` 省略時（通常送信）は従来と完全同一挙動（バイト同一の回帰テスト維持）。
  未知のノード id は 422 を返す（FastAPI バリデーション。フロントは既存のエラー表示に乗せる）。
- **バックエンド**: `origin_node` があるとき、campus_map の申告用ラベル
  （`ORIGIN_SELECT_LABELS`: フロントの selectionLabel と同一の解決可能な別名 —
  例: cafeteria→「カフェテリア（食堂）」、k→「共通施設棟（総合受付）」）を用いて
  **内部で**「現在地は{label}です。{message}」を合成し、これを従来どおり
  エージェント入力・DB content・4 ターン履歴に用いる（§7-3 の設計効果を維持）。
  あわせて user メッセージの `map_json` に `{"mode": "origin_select", "origin": {"node", "label"}}` を保存し、
  `GET /api/threads/{id}` で返す（既存の `map` 返却は role を問わない実装のため追加変更は最小）。
- **フロント表示**: `map.mode === "origin_select"` の user メッセージは、本文の代わりに
  **現在地チップ**（例: 「📍 現在地: カフェテリア（食堂）」）をユーザー側バブル位置に小さく表示する。
  ライブ送信時もローカルで同じチップを即時表示する（リロード後は map_json から同一描画）。
  合成文そのものは UI に出さない。
- タップ直後、選ばれたノードはカード上で**選択済み状態**として残す（§11-3 の意匠で表現）。
- **エージェント履歴のサニタイズ（v0.3 検収時追加・Fable 裁定）**: ask_origin ターンの
  assistant 定型文（「いまいる場所をマップでタップして…」）を、エージェントに渡す履歴に
  **そのまま入れない**。履歴組み立て時に `map_json.mode == "ask_origin"` の assistant メッセージの
  content を「（現在地の選択をお願いしました）」へ置換する。
  理由（検収実測）: キャンセル後に無関係の質問（「GI512はどこ？」）をすると、生成 LLM が
  履歴中の elicitation 文言を「未回答の質問」と誤認し、前の質問（「体育館に行きたい」）への
  回答を生成した。DB の保存内容・画面表示は変更しない（置換はエージェント入力のみ）。

### 11-3. カード UI の作り込み（デザインリード: Codex/GPT-5.6Sol）

FR-12 と同じ体制 — **デザイン判断は Sol 主導**、Fable は制約定義と検収のみ。
現行 MapCard は機能検収用の素朴な実装であり、「Campus Signal」の世界観
（`docs/UI_POLISH_V2.md`）に合う水準まで作り込む。

     既存カード類（出典折りたたみ・About モーダル・「最新へ戻る」ガラスボタン）と同じ家族に見えること。

  ②ask_origin 選択済み/キャンセル済み/履歴（非活性）③route（経路ハイライト＋ステップ）
