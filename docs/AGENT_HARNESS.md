# エージェントハーネス v2 仕様

- 版: v1.0（2026-07-11, Fable 起草）
- 背景（利用者フィードバック 2 件）:
  1. 「公式サイトやオープンキャンパス当日にご確認ください」型の**丸投げ回答を禁止**。その分ハーネス側で RAG と Web Search を粘り強く回す。
  2. RAG で取れない情報は**必ず Web Search で取りに行く**（現行のキーワードゲートは廃止）。
- 対象: `backend/app/agent/graph.py` の RealCampusAgent を v2 に置き換える。SSE スキーマ・`stream()` インターフェースは不変。

## 1. フロー（LangGraph）

```
analyze → retrieve → evaluate ─┬─(sufficient)──────────→ generate
                               └─(insufficient)→ web_search → evaluate（最大2周）
```

- **evaluate が不十分と判定したら無条件で web_search へ**。現行の `_needs_current_or_external_info`
  キーワードゲートは**削除**する。
- web_search → evaluate のループは**最大 2 周**。2 周しても不十分なら「持っている情報で最大限」generate へ。

## 2. 各ノード仕様

### analyze（LLM 呼び出し・軽量）
- 入力: 質問 + 直近履歴。出力 JSON: `{"retrieval_queries": ["...", "..."], "intent": "..."}`
- 検索クエリを**観点を変えて 2〜3 本**生成させる（例: 「学食」→「食堂 メニュー」「カフェテリア 営業時間」）。
- max_tokens ≤ 300。JSON パース失敗時は素の質問文 1 本にフォールバック。

### retrieve
- analyze の全クエリで Qdrant を検索（各 top_k=6）し、ID で重複排除してスコア降順マージ（上限 10 チャンク）。

### evaluate（LLM 呼び出し・軽量 — 従来のスコア閾値判定から置き換え）
- 入力: 質問 + 現在のコンテキスト（ナレッジ + これまでの Web 結果）。
- 出力 JSON: `{"sufficient": bool, "missing": "足りない情報の説明", "web_queries": ["...", "..."]}`
- プロンプトで判定基準を明示: 「**来場者に具体的な回答（数字・固有名詞・手順）を返せるか**。
  一般論しか言えない・重要な詳細が欠けるなら insufficient」。
- max_tokens ≤ 300。パース失敗時は `sufficient=true` として先へ進む（停止させない）。
- 補助としてスコア閾値 0.45 を「明らかな無関係」の足切りに残してよいが、主判定は LLM。

### web_search（強化）
- evaluate の `web_queries` を使用（無ければ質問文）。1 周目は `site:akita-pu.ac.jp` を付けたクエリを優先し、
  2 周目は制限なしの一般クエリ。各周 ddgs で最大 5 件取得。
- **スニペットだけで終わらせない**: 上位 3 URL の本文を httpx で取得（timeout 8s、失敗は握り潰して次へ）し、
  HTML から本文テキストを抽出（bs4。script/style/nav 除去）して**1 ページ 4,000 字まで**コンテキストに追加する。
- 取得したページは Source(type="web") として done.sources に載せる（実際にコンテキスト投入したもののみ）。
- 依存追加: `beautifulsoup4`（httpx は導入済み）。

### generate
- ストリーミング生成。システムプロンプトに以下を**明文で**入れる:
  1. **禁止**: 「公式サイトでご確認ください」「当日の学科紹介でご確認いただけます」等、
     **調べれば答えられる内容を利用者に丸投げする表現を回答の主内容にすること**。
  2. コンテキストにある情報は数字・固有名詞・手順まで**具体的に**盛り込む。省略しない。
  3. 検索を尽くしても根拠が無い場合のみ「現時点で確認できなかった」と正直に述べ、
     **何をどこまで調べたか**（学内ナレッジ・大学公式サイトの Web 検索）を一言添えた上で、
     補助情報として問い合わせ先を案内するのは可。
  4. 年度依存情報の年度明記・根拠外の内容やURLの捏造禁止・ロール別トーンは現行を維持。

## 3. ステップ実況（SSE）

- step の enum（analyze/retrieve/evaluate/web_search/generate）は**変更しない**（フロントのテーマが step キー）。
- 2 周目以降は同じ step でもテキストを変える: 例
  - web_search 2周目: 「別の観点で Web を調べています…」
  - evaluate 2回目: 「集めた情報を検証しています…」
- analyze・evaluate の LLM 呼び出し中もステータスは出しっぱなしでよい（追加イベント不要）。

## 4. LLM 呼び出し規約（モデル変更に伴う変更）

- モデルは `google/gemma-4-31B-it-qat-w4a16-ct`（ARCHITECTURE.md §5 改訂参照）。
- **Qwen 固有の `chat_template_kwargs: {"enable_thinking": false}` は送らない**こと（Gemma のテンプレートは
  受け付けない）。`extra_body` はモデル名に `qwen` を含む場合のみ付与する実装にする（設定で切替可能に）。
- analyze/evaluate は temperature 0.2・低 max_tokens、generate は temperature 0.7 目安。

## 5. 受け入れ基準（Fable が実測でレビュー）

1. 「サークルにはどんなものがありますか？」→ 団体名を列挙した具体回答（丸投げ表現なし）
2. 「オープンキャンパスでは何ができますか？」→ プログラム内容を具体列挙（「当日ご確認ください」を主内容にしない）
3. ナレッジに無い学内情報（例: 「学長は誰ですか」）→ web_search ステップが発火し、公式サイトから取得して回答
4. 学外・答えようのない質問（例: 「明日の天気」）→ 検索した上で、調べた範囲を添えた誠実な回答
5. 応答開始（最初の status）1 秒以内は維持、全体レイテンシは 60 秒以内目安
