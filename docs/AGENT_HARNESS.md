# エージェントハーネス仕様（v4 = v3 + Tavily 移行・柔軟再試行）

- 版: v4.0（2026-07-12, Fable 起草）。**v3 本文（下記）を基底とし、§V4 のデルタを適用する。矛盾時は §V4 が優先。**
- v4 の背景（利用者指示 2026-07-12）:
  1. Web Search プロバイダを ddgs（DuckDuckGo）から **Tavily** に移行する。API キーは `.env` に記載済み。
  2. 試行順序は **Agentic RAG → Agentic Search → Web Search**（v3 の逐次順を維持・明文化）。
  3. **「回答不可」を避ける**ため、evaluate 主導でローカル/Web の再試行を柔軟に行えるループにする。

## V4. デルタ仕様

### V4-1. TavilySearchProvider（ddgs 廃止）

- 新規 `backend/app/search/tavily.py` に `TavilySearchProvider` を実装。`backend/app/search/ddgs.py`・`ddgs` 依存・関連テストは削除（テストは Tavily 版に置換）。
- インターフェース（`SearchProvider` 抽象は踏襲・拡張）:
  `async def search(query: str, *, max_results: int = 3, include_domains: Sequence[str] | None = None) -> list[WebSearchResult]`
- 実装は **httpx 直叩き**（新規依存を増やさない。tavily-python は使わない）:
  - `POST https://api.tavily.com/search`、認証は `Authorization: Bearer <TAVILY_API_KEY>` ヘッダ
    （Tavily 現行 API の推奨形式。実装時に公式ドキュメントで最終確認し、異なれば body の `api_key` 形式に合わせる）。
  - body: `{"query", "max_results", "search_depth": "basic", "include_answer": false, "include_raw_content": true, "include_domains": [...]（指定時のみ）}`
  - タイムアウト 8 秒（既存の fetch と同じ）。
- レスポンスマッピング: `results[]` の `title`→title、`url`→url、`content`→snippet、
  `raw_content`（存在すれば）→ `WebSearchResult.text`（全文として保持。窓切り出しはハーネス側 §4.3）。
- `include_answer` は **false 固定**（Tavily の生成回答は使わない。回答生成の主導権と「ナレッジ優先」ルール（§5 ルール5）を守るため）。
- **エラーの飲み込み（回答不可回避の要）**: HTTP エラー・タイムアウト・JSON 不正・キー未設定は
  **例外を投げず `[]` を返す**（warning ログ）。0 件扱いになれば既存の 0 件リフォーミュレーション（§4.2）と
  諦めゲート（§5）がそのまま機能する。ハーネスを 500 で落とすことは決して無いこと。
- 設定: `Settings.tavily_api_key`（`os.getenv("TAVILY_API_KEY", "")`）。
  `.env` に記載のキー名が正（標準名 `TAVILY_API_KEY` を想定。実物が別名ならコードを .env に合わせる）。
  docker-compose.yml の backend `environment:` に `TAVILY_API_KEY: ${TAVILY_API_KEY:-}` を追加
  （compose はルートの `.env` を自動で補間に使う）。
- requirements.txt: `ddgs==9.14.4` を削除。`httpx==0.28.1` の pin は維持し、ddgs 由来のコメントを更新。

### V4-2. `site:` 制限の廃止 → `include_domains`

- v3 §4.1 の 1 周目 `site:akita-pu.ac.jp <query>` を廃止し、
  **1 周目は `include_domains=["akita-pu.ac.jp"]`** を渡す（サブドメイン cps.akita-pu.ac.jp 等も含まれる）。
  クエリ文字列自体は変形しない。2 周目以降はドメイン制限なし（v3 どおり「秋田県立大学」自動付与は維持）。
- 使用済みクエリの重複判定は **（正規化クエリ, ドメイン制限の有無）の組**で行う
  （同一文言でも 1 周目の制限付きと 2 周目の制限なしは別試行として許可）。
- 0 件リフォーミュレーション（v3 §4.2）は維持。「`site:` を外す」は「`include_domains` を外す」に読み替え。

### V4-3. raw_content の活用（focused extraction は維持）

- ページ本文は **Tavily の `raw_content` を第一ソース**とし、`raw_content` が無い結果だけ従来の httpx fetch を行う
  （fetch 回数が減る分レイテンシと取得失敗が減る）。
- どちら由来でも v3 §4.3 の focused extraction（キーワード窓 1,400 字）をそのまま適用する。
- フェッチ前 URL 選別（v3 §4.3 後段）も不変。

### V4-4. 柔軟な再試行ループ（回答不可の回避）

- **初回パスの順序は不変**: analyze → retrieve（Agentic RAG）→ search（Agentic Search）→ evaluate。
- evaluate の出力 JSON に `retrieval_queries` を追加（不足時のみ・最大 2・既出と別観点の言い換え）:

```json
{"sufficient": true|false, "missing": "...",
 "grep_keywords": ["..."], "retrieval_queries": ["..."], "web_queries": ["...", "..."]}
```

- ルーティングを優先度型に統一（`_route_after_evaluate` を置換）:
  1. `sufficient=true` → generate
  2. `grep_keywords` 非空 かつ search 追加ラウンド未消化（最大 1、v3 踏襲）→ **search**
  3. `retrieval_queries` 非空 かつ **re-retrieve 未消化（最大 1、v4 新規）** → **retrieve**
     （使用済み retrieval クエリを正規化して記録し、全て既出ならこの分岐はスキップ）
  4. Web ラウンド条件（v3 §4.1、最大 3）→ **web_search**
  5. generate（諦めゲート v3 §5 不変: retrieve・search（変種込み）・web 2 周以上を消化するまで
     「確認できなかった」系回答を出さない）
- retrieve 再実行時のステータス実況: `FOLLOWUP_STATUS_TEXTS["retrieve"] = "観点を変えて学内ナレッジを検索しています…"`。
  step enum・SSE スキーマの変更はなし。
- LLM 呼び出し上限: analyze 1 + evaluate 最大 6 + generate 1 = **8**（v3 は 7）。
  全体レイテンシ目標 60 秒以内は維持（V4-3 の fetch 削減分を充てる）。
- アームが例外で落ちた場合も**残りのアームは必ず実行**して generate に進む（v3 §5 踏襲・スキップ理由はログ）。
- 制御は引き続き `stream()` 内の逐次制御が正（Q-006 裁定）。LangGraph のノード定義は
  stream() と矛盾しない範囲で現状維持でよい（v4 で書き換え必須なのは stream() 側のみ）。

### V4-5. テスト・受け入れ基準（v3 §9 に追加）

- ユニットテスト:
  1. Tavily レスポンス → `WebSearchResult` マッピング（raw_content の text 反映含む）
  2. 1 周目のみ `include_domains` が付くこと・2 周目以降は付かないこと
  3. raw_content があるページは httpx fetch しないこと
  4. API エラー・タイムアウト・キー未設定 → `[]`（例外にならない）
  5. evaluate の `retrieval_queries` による re-retrieve（最大 1・使用済みクエリのスキップ）
  6. ルーティング優先度（search > retrieve > web_search > generate）
- 実測受け入れ（Fable が実施）:
  1. v3 §9 の全基準を維持（CPS 研の先生・学長名・サークル列挙・OC プログラム・天気の誠実回答）。
  2. ナレッジに無く Web にある情報が Tavily 経由で回答に入ること。
  3. `TAVILY_API_KEY` 未設定でも 500 にならず、ローカル 2 アーム＋誠実回答で完結すること。

# （基底）エージェントハーネス v3 仕様

- 版: v3.0（2026-07-12, Fable 起草）。v2.0（2026-07-11）を全面改訂。
- 背景（利用者フィードバック）:
  1. （v2 から継続）丸投げ回答の禁止。RAG と Web Search を粘り強く回す。
  2. （v2 から継続）RAG で取れない情報は必ず Web Search で取りに行く。
  3. **（v3 新規）Agentic RAG に加えて、grep 等をツールに持つ「Agentic Search」を追加する。**
  4. **（v3 新規）Web Search ハーネスの強化。** 実測の失敗例:
     「サイバーフィジカルシステム研究室の先生は誰ですか？」→「資料に記載がない」と回答。
- 対象: `backend/app/agent/graph.py` の RealCampusAgent を v3 に改訂。SSE の `stream()` インターフェースは不変、step enum に `search` を**追加**（§6）。

## 0. v2 の敗因分析（Fable 実測、2026-07-12）

失敗質問「サイバーフィジカルシステム研究室の先生は誰ですか？」について:

1. **ベクトル検索は固有名詞に弱い**。再構築後のナレッジ（76ファイル・434チャンク）には
   「サイバーフィジカル」を含む記述が4箇所ある（経営システム工学科・山口高康教授ほか）にもかかわらず、
   bge-m3 の top-5 に該当チャンクは入らない（正しい**ファイル**の別チャンクが2位・3位に来て、肝心の行は落ちる）。
   → 字句一致（grep）なら一発で当たる。**これが Agentic Search を追加する理由。**
2. **Web ページの先頭切り出しが情報を捨てる**。v2 はページ本文の**先頭 1,400 字**だけを使うため、
   教員一覧・研究室一覧のようにページ中盤〜末尾にある情報は、正しいページを取得できていても
   コンテキストに入らない。→ **キーワード出現位置を中心に窓を切る**（§4.3 focused extraction）。
3. **site: 制限クエリが 0 件でもそのまま終わる**。ddgs の `site:` 検索はニッチな日本語フレーズで
   0 件になりやすい。→ 0 件時の自動リフォーミュレーション（§4.2）。

補足の実測: ナレッジは公式サイトの最新転記（retrieved_at 付き）だが、Web 検索は古いページを拾い得る
（例: 学長名で 2011 年就任の旧学長ページがヒット）。→ 矛盾時はナレッジ優先ルール（§5 ルール5）。

## 1. 全体フロー

```
analyze（LLM: 意図・キーワード・検索クエリ）
  → retrieve（Agentic RAG: Qdrant ベクトル検索）   ┐ 並行実行してよい
  → search  （Agentic Search: knowledge/ 字句検索） ┘
  → evaluate（LLM: 十分か? 次アクションは?）
  → [不足なら] search/retrieve 追加ラウンド（evaluate の指示語で再 grep / 再検索）
  → [なお不足なら] web_search → evaluate（最大 3 周）
  → generate
```

- 制御は v2 と同じく**明示的な逐次制御**（LangGraph のノード定義は維持してよいが、
  `stream()` 内のループが正。Q-006 の裁定を継承）。
- ローカル 2 アーム（retrieve + search）を先に尽くし、それでも不足なら Web へ。
  **ローカル追加ラウンドは最大 1 回**（evaluate が新キーワード/新クエリを出した場合のみ）。
- **Web ラウンドは最大 3**（v2 は 2）。3 周目は「固有名詞が未解決」の場合のみ発火（§4.2）。
- LLM 呼び出し回数の上限: analyze 1 + evaluate 最大 5 + generate 1 = 7。
  レイテンシ目標は全体 60 秒以内を維持。

## 2. analyze（LLM・軽量）

- 入力: 質問 + 直近履歴。
- 出力 JSON（v2 から拡張）:

```json
{"intent": "...",
 "retrieval_queries": ["...", "..."],
 "keywords": ["サイバーフィジカルシステム研究室", "..."]}
```

- `keywords`: 質問中の**固有名詞・専門用語・レアな語**（研究室名・人名・制度名・建物名など）を
  そのまま抜き出す（言い換え禁止・最大 4 語）。一般語（「先生」「場所」等）は含めない。
- `retrieval_queries` は v2 と同じ（観点違い 2〜3 本）。
- max_tokens ≤ 300、temperature 0.2。JSON パース失敗時のフォールバック:
  `retrieval_queries=[質問文]`、`keywords=[]`（このとき §3.3 の自動変種生成が質問文から効く）。

## 3. Agentic Search（新設・step 名 `search`）

### 3.1 コーパスのロード

- `KNOWLEDGE_DIR` 環境変数（デフォルト: リポジトリの `knowledge/`。docker では
  `/app/knowledge` にリードオンリーマウント — §8）配下の `*.md` を対象とする
  （`SOURCES.md` と `ingest/` は除外）。
- 起動時（初回利用時の lazy load でも可）に全ファイルを読み、**ingest と同じ規則**
  （frontmatter をメタデータ、`##` 見出し単位でセクション分割）で
  `(file_id, title, category, source_urls, heading, section_text)` のリストをメモリに保持する。
  コーパスは約 200KB なので常駐で問題ない。
- ファイルの変更検知・再読込は不要（プロセス再起動で反映、で足りる）。

### 3.2 検索ツール `grep_sections(keywords) -> list[SectionHit]`

- 各キーワードを **NFKC 正規化 + 小文字化**した上で、全セクション本文（title・heading 含む）に
  対して部分一致検索する（正規表現は不要。固定文字列でよい）。
- スコアリング: ①ヒットした**異なりキーワード数**（多いほど上位）→ ②総ヒット回数 → ③本文が短い方。
- 上位 **4 セクション**まで返す。1 セクションは KnowledgeChunk 相当に変換して
  `knowledge_results` にマージする（`id` はベクトル検索と同じ規則で作り、**ID で重複排除**。
  同一セクションが両アームから来たらスコアの高い方を残し、**grep ヒットフラグを立てる**）。

### 3.3 キーワード変種の自動生成（0 件時のフォールバック・決定的処理）

キーワードが 1 件もヒットしない場合、次の順で変種を作り再検索する（LLM 不要・純 Python）:

1. **接尾辞除去**: 「研究室」「研究グループ」「講座」「学科」「先生」「教授」「について」を末尾から除去。
   例: サイバーフィジカルシステム研究室 → サイバーフィジカルシステム
2. **最長文字種ラン抽出**: **接尾辞除去後の語**の最長のカタカナ連続列・漢字連続列を取り出す。
   例: サイバーフィジカルシステム → （カタカナランそのもの）、知能メカトロニクス学科 → 知能メカトロニクス
3. それでも 0 件なら「ヒットなし」として evaluate に渡す（無理に緩めない。誤ヒットの方が害）。

**汎用語ガード（2026-07-12 受け入れテストで追加）**: 変種として接尾辞リストの語そのもの
（「研究室」「学科」等）や除去前の原語由来の汎用ランを出してはならない。
実測の失敗: 「サイバーフィジカルシステム研究室」の漢字ラン「研究室」が変種となり、
「研究室」を16回含む建物案内セクションが本命の CPS セクションを top-4 から押し出した。
ラン抽出の対象は接尾辞除去後の語のみとし、生成された変種が VARIANT_SUFFIXES に含まれる場合は捨てる。

### 3.4 ステータス実況

- step: `search`、テキスト: 「学内資料を全文検索しています…」
- 追加ラウンド時: 「別のキーワードで学内資料を調べています…」

## 4. Web Search（強化）

### 4.1 ラウンド構成（最大 3 周）

| 周 | クエリ方針 |
|---|---|
| 1 | `site:akita-pu.ac.jp <query>`（公式サイト限定） |
| 2 | 制限なし。クエリに「秋田県立大学」が無ければ自動付与 |
| 3 | **未解決の固有名詞がある場合のみ**。エンティティ変種クエリ（§4.2）＋ evaluate の再指示クエリ |

- 各周: evaluate の `web_queries`（無ければ質問文）を使用、最大 5 件取得・上位 3 URL の本文取得は v2 踏襲。

### 4.2 クエリ・リフォーミュレーション（決定的処理 + LLM 併用)

- **0 件検知**: ある周の全クエリの検索結果が合計 0 件だったら、同じ周のうちに 1 回だけ
  自動変形して再検索する: `site:` を外す / キーワード変種（§3.3 と同じ規則）に置き換える。
- **エンティティ変種クエリ**（3 周目用・自動生成）: analyze の `keywords` のうちナレッジ・Web で
  未解決のものについて `「<entity> 秋田県立大学」`、`「<entity(接尾辞除去)> 秋田県立大学 教員」` を生成。
- evaluate は各周で `web_queries` を**前の周と違う観点**で出し直す（プロンプトで明示）。
  ハーネス側でも**使用済みクエリを記録**し、同一クエリの再実行をスキップする。

### 4.3 focused extraction（ページ本文の切り出し改善 — v3 の要）

- ページ本文はまず**全文**を抽出し（bs4、script/style/nav 除去は v2 踏襲）、その後:
  1. analyze の `keywords`（＋その変種）のいずれかが本文中に出現するなら、
     **最初の出現位置を中心に前後合わせて `MAX_WEB_PAGE_CHARS`(=1400) 字の窓**を切り出す
     （窓の先頭に `…` を付けて途中開始を明示）。複数キーワードが離れて出現する場合は
     最多キーワードを含む窓を選ぶ（実装は「出現位置ごとに窓を作りキーワード数を数える」程度の単純さでよい）。
  2. キーワードが出現しないページは従来どおり先頭 1,400 字。
- **フェッチ前の URL 選別**: 検索結果の title+snippet にキーワードを含むものを優先して上位 3 件を選ぶ
  （含むものが 3 件未満なら残りはスコア順で埋める）。

## 5. generate

v2 のルール 1〜4 を維持し、以下を追加:

- **ルール5（情報源の優先順位）**: 「学内ナレッジ」は公式サイトの最新転記である。
  Web 検索結果と学内ナレッジが矛盾する場合は**学内ナレッジを優先**し、
  Web の古いページ（過去の役職者・旧年度情報など）に引きずられないこと。
- **調査ログの明示**: ルール3（正直な不明回答）の際に添える「何をどこまで調べたか」は、
  ハーネスが実際の実行履歴から生成した 1 行（例:
  `調査ログ: 学内ナレッジ検索2回・全文検索(キーワード: …)・Web検索3回(クエリ: …)`）を
  プロンプトに渡し、それを根拠に書かせる。捏造防止。
- **コンテキスト詰め込み順**（予算逼迫時の優先順。2026-07-12 改訂）:
  1. **grep ヒットのセクション**（字句一致 = 精度最高）
  2. **高関連度のベクトルチャンク**（score ≥ `HIGH_RELEVANCE_CONTEXT_THRESHOLD = 0.60`、スコア順）
  3. Web 結果（focused extraction 済み）
  4. 残りのベクトルチャンク（スコア順）
  ※ 旧順序（grep → Web → ベクトル全部）は、evaluate が誤って不足判定し Web を回した場合に、
  正解を含む高スコアの学内ナレッジ（一次情報）が Web の薄いページに予算を奪われて
  generate に届かない実測欠陥があった（CPS 研出展質問で再現、2026-07-12 検収）。
  ルール5（ナレッジ優先）とも整合させ、高関連度ナレッジを Web より先に確保する。
- **ルール6（会話履歴の常時保持）**: 会話履歴（`_format_history` 済み、
  最大 `MAX_HISTORY_MESSAGES` 件・各 `MAX_HISTORY_CHARS` 字）は生成プロンプトに**常に含める**。
  プロンプトが予算超過しても履歴を丸ごと落とすことは禁止（マルチターン会話が成立しなくなるため）。
  縮退は次の順で行う:
  1. 根拠コンテキストを縮小して再組み立て
     （コンテキスト予算 = プロンプト予算 − system − **履歴** − user 骨格）。
  2. 残余コンテキスト予算が `MIN_GENERATION_CONTEXT_TOKENS = 384`（推定トークン）を下回る場合、
     履歴の**件数は維持**したまま各メッセージの文字上限を段階的に縮めて根拠予算を確保する
     （500 → 250 → 120 字。下限 120 字。下限でも届かなければ残余予算で妥協し、根拠 0 も許容）。
  3. それでも超過する場合は最後の user メッセージ（根拠部分）を切り詰める。
  この順序は強制コンテキスト予算パス（実トークン計測後の再構築・リトライ）にも同様に適用するが、
  強制パスでは実測由来の予算を 384 へ単純に押し上げてはならない（実超過から回復不能になる）。
  強制パスでの 384 確保は履歴短縮で捻出した推定トークン分の加算のみ
  （最終予算 = min(残余推定予算, 強制予算 + 履歴短縮で浮いた分)）。
  evaluate モードのプロンプト構築はこの最低根拠保証の対象外（詳細は QUESTIONS.md Q-011 裁定）。

### 諦めゲート（丸投げ・不明回答の機械的抑制）

`sufficient=false` のまま全ラウンドを使い切った場合のみ「確認できなかった」系の回答を許可する。
具体的には、generate プロンプトに調査ログを渡すのは**次の全てを満たすとき**だけ:

- retrieve 実行済み、search（変種込み）実行済み、web_search を 2 周以上実行済み。

それ以前に evaluate が `sufficient=true` を返せば通常回答。途中でエラーにより
アームをスキップした場合は、残りアームを実行してから generate に進む（スキップ理由はログへ）。

## 6. ステップ実況（SSE）

- step enum: `analyze | retrieve | search | evaluate | web_search | generate`（**search を追加**）。
- ARCHITECTURE.md §3 の SSE スキーマも同時に更新すること。
- **フロントエンド**: LoadingSpinnerV5 のステップテーマに `search` を追加する
  （retrieve 系の亜種として新規テーマ 1 色を割り当て。実装は既存テーマ定義に 1 エントリ追加）。
  併せて**未知の step が来ても既定テーマにフォールバック**する防御を入れる（今後の enum 追加に耐える）。
- ステータステキスト:

| step | 初回 | 2回目以降 |
|---|---|---|
| search | 学内資料を全文検索しています… | 別のキーワードで学内資料を調べています… |
| web_search | Webで最新情報を確認しています… | 別の観点でWebを調べています…（3周目: さらに手掛かりを探しています…） |

他は v2 のまま。

## 7. evaluate（v2 から拡張）

- 出力 JSON:

```json
{"sufficient": true,
 "missing": "...",
 "grep_keywords": ["..."],
 "web_queries": ["...", "..."]}
```

- `grep_keywords`: 不足時、**ナレッジ内をこの語で字句検索すれば埋まりそうな固有名詞**があれば返す
  （最大 3。無ければ空配列）。ハーネスはこれが非空かつローカル追加ラウンド未消化なら
  web より先に search を 1 回だけ再実行する。
- 判定基準の文言は v2 を維持（「具体的な回答（数字・固有名詞・手順）を返せるか」）した上で、
  **値要求チェックを追加（2026-07-12 受け入れテストで追加）**: 質問が特定の値
  （人名・数値・日時・場所・URL）を求めている場合、**その値そのものがコンテキストに無ければ必ず insufficient**。
  周辺情報（関連する研究内容の説明など）があっても値が無ければ sufficient にしない。
  実測の失敗: 「〜研究室の先生は誰？」に対し、教員名が無いのに CPS 研究の説明だけで
  sufficient=true と判定され、Web 検索（1周目で正解 cps.akita-pu.ac.jp が出る）に到達しなかった。
- パース失敗時 `sufficient=true` で先へ進む（停止させない）のも v2 踏襲。
- コンテキスト圧縮: evaluate に渡す各根拠は 400 字要約（v2 踏襲）。grep ヒットセクションは
  **マッチ行を中心に** 400 字（先頭からではない — §4.3 と同じ思想）。

## 8. 設定・環境

- 新規環境変数: `KNOWLEDGE_DIR`（デフォルト `knowledge/` をリポジトリルートから解決。

## 9. 受け入れ基準（Fable が実測でレビュー）

1. **「サイバーフィジカルシステム研究室の先生は誰ですか？」→ 固有名詞入りの具体回答**
   （ナレッジの grep ヒットから山口高康教授（経営システム工学科・持続可能マネジメント講座）等、
   コーパス記載の事実で答える。「資料に記載がない」は不合格）。
2. v2 基準の継続: サークル列挙 / OC プログラム列挙 / 学長名（**ナレッジ優先で小林淳一**になること
   — Web の旧学長ページに引きずられたら不合格）/ 明日の天気（誠実回答）。
3. ナレッジに無く Web にある情報 → web_search 発火・focused extraction でページ中盤の情報も拾えること。
4. 応答開始（最初の status）1 秒以内、全体 60 秒以内目安。
5. ユニットテスト: キーワード変種生成（§3.3）・grep スコアリング（§3.2）・focused extraction（§4.3）・
   詰め込み順（§5）・諦めゲート（§5）・SSE に search step が出ること。既存テストは全て green を維持。
